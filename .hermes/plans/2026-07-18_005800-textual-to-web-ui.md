# Research Hub: Textual TUI → Web UI Conversion

> **Goal:** Replace the Textual terminal dashboard with a Flask+Jinja+HTMX web UI that preserves every backend capability (kanban, profiles, settings, memories, phase control) while making the tool inspectable, steerable, and screenshottable from any browser.

**Architecture:** The existing `hub.py` is a clean pure-Python backend (SQLite + `hermes kanban` subprocess + memory file writes). The Textual `dashboard.py` is a separate presentation layer that calls it. **We add `webapp.py` as a new presentation layer alongside dashboard.py** — hub.py stays untouched, dashboard.py stays as a fallback.

**Tech Stack:** Flask + Jinja2 templates + HTMX (for live polling without writing JS). No build step. Single-file Flask app to start.

**Preserved capabilities (the user's must-haves):**
1. Hermes kanban — `hub.py`'s `create_kanban_board`, `create_kanban_task`, `poll_idea_phase` all unchanged
2. Hermes profiles — referenced by name in config.yaml; web UI lists/displays them
3. Research goals/restrictions — `setting.md` editing (goal, task, constraints, success criteria, team settings)
4. Memories/souls — `write_agent_memory` unchanged; web UI surfaces what was written
5. Phase control — `setup_idea_phase`, `poll_idea_phase`, `get_idea_status` unchanged; web UI triggers + displays

## Tasks (in execution order)

### Task 1: Flask skeleton + base template
- Create `webapp.py` with Flask app, import hub.py
- `/` route: list projects, redirect to first or show empty state
- `templates/base.html`: sidebar (projects list + profiles link) + main content block
- `static/style.css`: dark theme matching the Textual look
- Run on localhost:5000, verify it serves

### Task 2: Project detail view (read-only)
- `/project/<id>` route
- Display project name, goal, description, status
- Show phases list (read-only)
- Show rounds table for idea phase (read-only snapshot from `get_idea_status`)

### Task 3: Settings editor (research goals/restrictions)
- `/project/<id>/settings` GET: show `setting.md` content in a textarea
- POST: save edited content back to `setting.md`
- This is where the user specifies goal, task, constraints, success criteria

### Task 4: Agent panel (profile selection + phase setup)
- `/project/<id>/agents` GET: show form with dropdowns for proposer/critic/manager (populated from config.yaml agents) + max_rounds input
- POST: calls `hub.setup_idea_phase(pid, proposer, critic, manager, max_rounds)`
- Success: flash message + redirect back to project view
- Surface errors clearly (e.g., profile doesn't exist, kanban create failed)

### Task 5: Progress panel with HTMX live polling
- `/project/<id>/progress` partial route: renders rounds table
- `hx-trigger="every 2s"` on the panel div
- Each poll calls `hub.poll_idea_phase(pid)` then re-renders table
- Status icons: ⏳ pending, 🔄 running, ✅ completed, ❌ failed
- Show elapsed time per round

### Task 6: Activity log panel
- `/project/<id>/activity` partial route: last N log events
- `hx-trigger="every 3s"` to refresh
- Logs source: phase state changes, round completions (derived from DB timestamps)

### Task 7: Profiles view
- `/profiles` route: lists all agents from config.yaml
- For each: profile name, display name, role, model, and a "view memory" link
- `/profiles/<name>/memory`: shows `~/.hermes/profiles/<name>/memories/MEMORY.md` (read-only display)

### Task 8: Round output viewer
- `/project/<id>/round/<num>/<role>`: displays the output markdown for a specific round/role
- Reads from `proj_dir/phases/00-idea/<role>/round-NN.md`
- Renders markdown to HTML

### Task 9: Browser screenshot verification + polish
- Start Flask app (background)
- Use browser capture to screenshot the running UI
- Send to user for review
- Fix visual issues, iterate

## Risks / tradeoffs

- **Keeping dashboard.py around** — small duplication of intent, but provides fallback and costs nothing. Can deprecate later if web UI wins.
- **HTMX polling vs SSE/websockets** — polling is simpler and fine for a single-user tool with 2s cadence. SSE would be cleaner but adds complexity.
- **No auth** — this is a local single-user tool. If deployed later, add auth then.
- **Subprocess blocking** — `setup_idea_phase` calls `hermes kanban create` synchronously, which can take a second or two. Run in a thread or show a spinner. For v1, accept the brief block.

## Out of scope for v1
- New phases beyond "00-idea" (the only one currently implemented)
- Editing config.yaml agents from the UI (edit the file directly for now)
- Multi-user / auth
- Deployment (local only first; Render later if wanted)
