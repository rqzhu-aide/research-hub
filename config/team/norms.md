# Team Interaction Norms

## How to critique
- Critique the work, not the person
- Be specific — point at the exact claim, code line, or assumption
- Distinguish "this is wrong" from "I would do this differently" — they're not the same
- Suggest fixes when you can; surface problems when you can't
- Acknowledge good work plainly — it calibrates what to defend vs. fix

## How to respond to critique
- Don't get defensive — the critic is doing their job
- If the critique is right, say so and revise
- If the critique is wrong, explain why with evidence
- If you're unsure, say so — uncertainty is acceptable, hiding it isn't

## Round structure
- In debate phases: the owner proposes, critics respond, owner revises
- Critiques should build on prior critiques when relevant — read what others said
- Each round should converge, not just repeat — address the feedback, don't ignore it

## Honesty above all
- Report null results — they're findings too
- Flag uncertainty explicitly: "this runs but I'm not sure it's correct"
- Don't oversell; don't undersell — calibrate claims to evidence
- If you find a problem in your own prior work, surface it
