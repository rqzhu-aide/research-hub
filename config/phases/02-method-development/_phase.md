# Phase: Method Development

## Goal
Propose and refine concrete method(s) for the project — translate the
literature review's findings into a specific, actionable approach that the
project will develop and validate.

## Gating
**Requires Phase 01 (Literature Review) to be complete.** The lead reads the
literature review summary before composing directives — the method must be
grounded in what the literature showed, not invented in a vacuum.

## Pattern
**Debate.** Members propose competing approaches from their own lens in round 1,
then critique and refine each other's proposals in subsequent rounds. The lead
facilitates convergence — either a single agreed method emerges, or the trade-offs
between options are articulated clearly enough for the user to decide.

This is NOT parallel (independent scans) or sequential (handoff pipeline).
Debate means members MUST read each other's work between rounds and respond to it.

## Folder
All outputs land in `ideas/run/NN/`:
- `round-01/<role>.md`, `round-02/<role>.md`, … — per-round proposals/critiques
- Summary goes to `phase-summaries/02-method-development.html` (not the run folder)

## Files in this folder
- `_lead.md` — how the research_lead runs a debate. **Read first if you are the lead.**
- `theorist.md`, `research_lead.md`, `data_scientist.md` — member task files (the protocol each member follows when proposing/critiquing).

## What success looks like
By the end of this phase, the project should have:
1. A concrete method proposal (mathematical formulation + algorithm sketch)
2. Clear positioning vs. prior art (what's new, what's borrowed)
3. Identified risks and open design questions
4. A recommendation: pursue this method, or choose between alternatives

If the team cannot converge, that's a valid outcome — but the trade-offs must be sharp enough for the user to make the call.
